
# Credential abuse detection — full implementation blueprint

## 1. Purpose and scope

**Goal:** detect and respond to **misuse of credentials**—stolen, replayed, shared, brute‑forced, or abused by insiders—across all identity surfaces:

- human users  
- service accounts  
- API keys / tokens  
- machine identities  

Outputs: high‑fidelity alerts, risk score updates, and optional automatic protections (step‑up auth, lockouts, geo/IP blocks).

---

## 2. High‑level architecture

**Core components:**

- **Signal Ingestor:** consumes authN/authZ events from the identity layer and apps.  
- **Feature & Pattern Engine:** builds per‑identity and per‑credential behavior profiles.  
- **Detection Engine:** rules + ML for credential abuse patterns.  
- **Protection Orchestrator:** executes mitigations (lock, step‑up, IP block, token revoke).  
- **Alert & Case Connector:** pushes alerts into IR/case systems and fusion center.

**Data sources:**

- Identity auth events (`login_success`, `login_failure`, `mfa_challenge`, `token_use`).  
- Application logs (API key usage, OAuth client usage).  
- Network context (IP, ASN, geo, device fingerprint).  
- Threat intel (known bad IPs, credential dumps).

---

## 3. Data model

### 3.1 Auth event enrichment

You already have `auth_events`; extend with credential‑centric fields:

```sql
ALTER TABLE auth_events
ADD COLUMN credential_type TEXT,      -- 'password','webauthn','api_key','token'
ADD COLUMN credential_id   TEXT,      -- hashed identifier
ADD COLUMN client_id       TEXT,      -- app / OAuth client
ADD COLUMN asn             INT,
ADD COLUMN device_fingerprint TEXT;
```

### 3.2 Credential abuse alerts

```sql
CREATE TABLE credential_abuse_alerts (
  id              UUID PRIMARY KEY,
  identity_id     UUID,
  credential_type TEXT,
  credential_id   TEXT,
  abuse_type      TEXT, -- 'bruteforce','stuffing','impossible_travel','shared_use','suspicious_api_key_use', etc.
  score           NUMERIC(5,2),
  details         JSONB,
  protection_actions JSONB,
  created_at      TIMESTAMPTZ DEFAULT now(),
  case_id         UUID
);
```

## 4. Feature & pattern engine

### 4.1 Per‑identity baselines

For each identity:

- **Normal IP ranges / ASNs**  
- **Normal geos**  
- **Typical login times**  
- **Typical devices / fingerprints**  
- **MFA usage patterns**  

Store as compact profiles:

```json
{
  "identity_id": "b1f0-...",
  "ip_clusters": ["198.51.100.0/24", "203.0.113.0/24"],
  "asn_clusters": [64500, 64501],
  "geo_distribution": { "US": 0.9, "CA": 0.1 },
  "hour_histogram": { "8": 0.2, "9": 0.3, "10": 0.2, "11": 0.1, "14": 0.2 },
  "device_fingerprints": ["dev:abc", "dev:def"]
}
```

### 4.2 Per‑credential baselines

For API keys / tokens:

- typical call rate  
- typical endpoints  
- typical source IPs / ASNs  
- typical time windows

## 5. Detection engine

### 5.1 Rule‑based detectors (must‑have)

**A. Brute force / password spraying**

- Many failed logins for one account from one IP / many IPs.  
- Many accounts targeted from a single IP.

**B. Credential stuffing**

- Multiple accounts accessed from same IP/ASN with similar patterns.  
- Sudden spike in failures followed by successes.

**C. Impossible travel**

- Successful logins from distant geos within short time window.

**D. Shared credential use**

- Same credential used from multiple geos / ASNs / devices concurrently.

**E. Suspicious API key use**

- API key used from new geo/ASN with high volume or new endpoints.

Example rule definition (YAML):

```yaml
id: "rule_bruteforce_single_account"
type: "window_count"
window: "5m"
condition:
  event_type: "login_failure"
group_by: ["identity_id"]
threshold:
  count: 20
abuse_type: "bruteforce"
severity: "high"
```

### 5.2 ML‑assisted detectors

- **Anomaly scoring** on login context (IP, geo, device, time) vs baseline.  
- **Sequence models** for login sequences (fail/fail/success from new IP).  
- **Clustering** of IPs/ASNs exhibiting credential‑stuffing behavior.

Simple anomaly score example:

```ts
interface LoginContext {
  ipNovelty: number;      // 0–1
  geoNovelty: number;     // 0–1
  timeNovelty: number;    // 0–1
  deviceNovelty: number;  // 0–1
  failedLoginsLastHour: number;
}

function credentialAbuseScore(ctx: LoginContext): number {
  let score = 0;
  score += ctx.ipNovelty * 0.3;
  score += ctx.geoNovelty * 0.2;
  score += ctx.timeNovelty * 0.1;
  score += ctx.deviceNovelty * 0.2;
  if (ctx.failedLoginsLastHour > 5) score += 0.2;
  return Math.min(1, score);
}
```

## 6. Streaming detection pipeline

### 6.1 Event flow

1. Auth event arrives (login success/failure, token use).  
2. Enrich with IP → ASN/geo, device fingerprint, threat intel.  
3. Fetch identity + credential baselines.  
4. Compute features + anomaly score.  
5. Evaluate rules + ML thresholds.  
6. If triggered → create `credential_abuse_alert` and call Protection Orchestrator.

### 6.2 Pseudo‑TypeScript

```ts
async function handleAuthEvent(event: AuthEvent) {
  const enriched = await enrichEvent(event);           // geo, ASN, device, TI
  const baseline = await loadBaseline(event.identity_id, event.credential_id);
  const features = buildFeatures(enriched, baseline);
  const score = credentialAbuseScore(features);

  const triggeredRules = evaluateRules(enriched, features);

  if (score >= 0.8 || triggeredRules.length > 0) {
    const abuseType = classifyAbuse(triggeredRules, features);
    const actions = await planProtections(abuseType, enriched, score);

    const alertId = await insertCredentialAbuseAlert({
      identity_id: event.identity_id,
      credential_type: event.credential_type,
      credential_id: event.credential_id,
      abuse_type: abuseType,
      score,
      details: { enriched, features, triggeredRules },
      protection_actions: actions
    });

    await executeProtections(actions, alertId);
  }
}
```

## 7. Protection orchestrator

### 7.1 Possible actions

- **Identity‑level:**
  - Force password reset.  
  - Require MFA on next login.  
  - Immediate session revocation.  
  - Temporary account lock.  

- **Network‑level:**
  - Block IP/ASN (short‑term).  
  - Rate‑limit from suspicious IPs.  

- **Credential‑level:**
  - Revoke API key / token.  
  - Rotate secrets.  

### 7.2 Policy‑driven protections

```yaml
id: "protect_high_score_credential_abuse"
match:
  abuse_type: ["bruteforce","stuffing","impossible_travel","shared_use"]
  min_score: 0.8
actions:
  - type: "revoke_sessions"
  - type: "require_mfa_next_login"
  - type: "notify"
    channel: "secops"
```

## 8. Example abuse scenarios

### 8.1 Credential stuffing from botnet

- Many accounts targeted from same ASN.  
- High failure rate, then some successes.  
- Rule: `rule_bruteforce_multi_account` triggers.  
- Actions: block ASN for 1h, require MFA for affected accounts, alert SecOps.

### 8.2 Stolen API key used from foreign cloud

- API key normally used from `US` on `ASN 64500`.  
- Suddenly used from `EU` on `ASN 64550` with high volume.  
- Anomaly score > 0.9.  
- Actions: revoke key, open case, notify service owner.

### 8.3 Shared admin credentials

- Same admin credential used from two distant geos within 10 minutes.  
- Baseline shows normally single device/geo.  
- Abuse type: `shared_use`.  
- Actions: force rotation, require individual accounts, open governance case.

## 10. Integration points

- **Zero‑Trust Identity Layer:**  
  Uses risk scores and can enforce MFA/lockouts based on abuse alerts.

- **Threat Hunting Engine:**  
  Abuse alerts become seeds for deeper hunts.

- **APT Behavior Models:**  
  Credential abuse is a strong signal of compromise feeding APT scoring.

- **Fusion Center:**  
  Correlates credential abuse with physical/influence indicators (e.g., targeted spear‑phishing campaigns).
